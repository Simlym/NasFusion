# 常见问题

## 安装与配置

### Q：首次同步需要多长时间？

**A：** 取决于 PT 站点的资源量和网络速度。

- MTeam 约有数十万条资源，全量同步约需 **20-60 分钟**
- 之后的增量同步只获取最新页，通常 **1-5 分钟**完成
- 建议首次同步设置较大的「同步页数」以获取完整数据

---

### Q：TMDB API Key 如何申请？

**A：** 按以下步骤申请：

1. 访问 [themoviedb.org](https://www.themoviedb.org/) 注册账号
2. 进入 **账号设置 → API → 申请 API 密钥**
3. 类型选择「个人（非商业用途）」
4. 填写简单描述（如"个人 NAS 媒体管理"）
5. 即可获得 API 密钥（v3 auth）

TMDB API 完全免费，个人用途无限速。

---

### Q：不配置 TMDB，系统能用吗？

**A：** 可以基本使用，但以下功能会受影响：

- ❌ 资源识别（无法关联元数据）
- ❌ 元数据刮削（无海报、简介等）
- ❌ 订阅创建（需要 TMDB 搜索剧集）
- ✅ 资源同步和搜索（正常可用）
- ✅ 手动下载（正常可用）

强烈建议配置 TMDB API Key。

---

### Q：Docker 和手动部署有什么区别？

**A：** 建议大多数用户使用 **Docker 部署**：

| | Docker | 手动部署 |
|--|--------|----------|
| 部署难度 | 简单（5 分钟） | 复杂（需配置 Python/Node 环境） |
| 隔离性 | 好 | 依赖系统环境 |
| 更新 | `docker compose pull && up -d` | 需手动操作 |
| 适合人群 | 大多数用户 | 有开发经验的用户 |

---

### Q：如何修改默认端口？

**A：** 编辑 `docker-compose.yml`，修改端口映射：

```yaml
services:
  frontend:
    ports:
      - "8080:80"    # 将前端改为 8080 端口

  backend:
    ports:
      - "9000:8000"  # 将后端改为 9000 端口
```

修改后执行 `docker compose up -d` 重启。

## PT 站点

### Q：支持哪些 PT 站点？

**A：** 目前内置 **MTeam（馒头）** 适配器，功能完整。

其他站点：
- 基于 NexusPHP 框架的站点（大多数国内 PT）：适配器开发中
- 其他站点：可通过开发自定义适配器接入

---

### Q：站点连接测试失败怎么办？

**A：** 按以下步骤排查：

1. **验证 API Key**：登录 PT 站点，确认 API Key 仍然有效（未过期、未被重置）
2. **检查代理**：如果站点需要代理访问，确认代理配置正确
3. **检查网络**：确认 NasFusion 所在的服务器能够访问站点域名
4. **查看日志**：进入 **任务调度 → 站点健康检查 → 历史日志** 查看详细错误

---

### Q：同步资源时显示 "API 频率限制"？

**A：** PT 站点对 API 调用频率有限制。解决方案：

- 适当增大同步间隔（如从 1 小时改为 6 小时）
- 减少每次同步的页数
- 如果多个站点同时同步，错开各站点的同步时间

---

## 订阅与下载

### Q：创建订阅后，为什么一直没有自动下载？

**A：** 可能原因：

1. **资源未同步**：检查是否有资源同步任务在运行，手动触发一次同步
2. **订阅匹配条件太严**：检查质量偏好、大小限制等是否过于严格
3. **站点没有该资源**：某些小众剧集在 PT 站点可能资源稀少
4. **下载器未配置**：确认 **设置 → 下载器** 中已添加并测试连接成功

在 **订阅详情** 中查看「候选资源」，若有候选但未触发下载，会有具体原因说明。

---

### Q：如何下载一部完整的电影/剧集合集？

**A：** 两种方式：

**方式一：直接下载**
1. 在 **资源搜索** 中搜索该资源
2. 找到整季/合集资源（标题通常含 "Complete" 或 "全集"）
3. 直接点击下载

**方式二：订阅**
1. 创建订阅，选择所有季度
2. 手动触发资源同步
3. 系统自动匹配并下载（可能分多个任务）

---

### Q：下载完成后文件在哪里？

**A：** 取决于配置：

- **qBittorrent 原始目录**：你在 qBittorrent 中设置的下载保存路径
- **整理后目录**：`MEDIA_ROOT` 配置的路径下（如 `/media/电影/`）

如果文件整理正常工作，你会在媒体库根目录下找到整理好的文件，Jellyfin/Plex 也会自动刷新显示。

---

### Q：硬链接模式要求什么条件？

**A：** 硬链接要求 **源文件和目标文件在同一文件系统**（同一磁盘分区）。

- ✅ qBittorrent 下载目录：`/downloads`，媒体库：`/media`（同一磁盘）
- ❌ qBittorrent 在机械盘，媒体库在固态盘（不同文件系统，硬链接失败）

跨磁盘时请改用 **软链接** 或 **复制** 模式。

Docker 环境下，确保两个目录挂载到同一宿主机磁盘：

```yaml
volumes:
  - /data/downloads:/downloads
  - /data/media:/media        # 同一 /data 盘，硬链接可用
```

## 媒体服务器

### Q：Jellyfin 没有自动刷新新资源？

**A：** 排查步骤：

1. 确认 **设置 → 媒体服务器** 中连接测试成功
2. 检查文件整理是否正常（文件是否已出现在媒体库目录）
3. 在 NasFusion 的下载任务详情中，查看是否有「通知媒体服务器」步骤的错误
4. 手动在 Jellyfin 中触发一次「扫描媒体库」，确认文件路径正确

---

### Q：观看历史同步有延迟？

**A：** 正常现象。观看历史同步默认每天执行一次（凌晨 3 点），不是实时同步。

如需立即同步，进入 **任务调度** 手动触发「观看历史同步」。

---

## 通知

### Q：Telegram 通知没有收到？

**A：** 排查步骤：

1. 在 **设置 → 通知 → Telegram** 中点击「发送测试通知」，看是否成功
2. 检查 Bot Token 格式（应为 `数字:字母`）
3. 确认 Chat ID 正确（与 Bot 对话后通过 `@userinfobot` 获取）
4. 确认已给 Bot 发过消息（Telegram 要求用户先主动联系 Bot 才能接收消息）
5. 如果 Telegram 在你所在地区受限，配置代理

---

### Q：能同时配置多个通知渠道吗？

**A：** 可以。Telegram 和邮件可以同时启用，系统会向所有启用的渠道推送通知。

你也可以针对不同事件类型选择不同渠道：
- 重要事件（下载完成、站点异常）→ Telegram
- 日常摘要（同步完成）→ 邮件

## 存储与数据库

### Q：SQLite 和 PostgreSQL 如何选择？

**A：**

- **个人使用、单用户**：SQLite 完全够用，零配置
- **多用户、高并发写入**：建议使用 PostgreSQL
- **NAS 上部署**：SQLite 更简单，推荐

切换数据库时需要重新初始化，历史数据无法直接迁移（建议重新同步）。

---

### Q：数据库文件在哪里，如何备份？

**A：** SQLite 数据库文件位于：

```
backend/data/nasfusion.db
```

Docker 部署时，对应宿主机路径是你在 `docker-compose.yml` 中挂载的位置。

简单备份命令：

```bash
# 备份
cp backend/data/nasfusion.db backup/nasfusion_$(date +%Y%m%d_%H%M).db

# 或者停止服务后备份（确保数据完整）
docker compose stop backend
cp backend/data/nasfusion.db backup/nasfusion.db
docker compose start backend
```

---

### Q：如何完全重置系统？

**A：** 删除数据库文件并重启：

```bash
docker compose down
rm backend/data/nasfusion.db
docker compose up -d
```

重启后数据库会自动重新初始化，需要重新配置站点、订阅等。

## 开发相关

### Q：如何添加对新 PT 站点的支持？

**A：** 继承 `BasePTSiteAdapter` 实现适配器类，参考 `backend/app/adapters/pt_sites/mteam.py`。

详细步骤：
1. 在 `backend/app/adapters/pt_sites/` 下创建新文件
2. 继承 `BasePTSiteAdapter`，实现 `test_connection`、`fetch_resources` 等方法
3. 在适配器注册表中注册新类型

如果你完成了新适配器，欢迎[提交 Pull Request](https://github.com/Simlym/NasFusion)！

---

**A：** 启动后端后访问：

- **ReDoc**：`http://localhost:8000/redoc`

两个文档都支持直接在浏览器中测试 API。
